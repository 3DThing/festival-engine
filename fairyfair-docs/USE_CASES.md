## fairyfair.ru — Посетитель

### UC-V-01 · Покупка билета 

| | |
|---|---|
| **Актор** | Любой пользователь |
| **Триггер** | Хочет попасть на фестиваль |
| **Предусловие** | Продажа билетов открыта |
| **Постусловие** | Билет куплен, QR-код отправлен на email |

#### Путь A: Без регистрации

**Основной поток:**
1. Нажимает «Купить билет»
2. Выбирает тип и дату
3. Вводит ФИО и email
4. Даёт согласие на обработку персональных данных
5. Переходит к оплате (YooKassa / Тинькофф)
6. Завершает оплату — получает номер заказа
7. Получает QR-код билета и номер заказа на email
8. Система предлагает создать аккаунт (по желанию)

**Альтернативные потоки:**
- Оплата не прошла — билет не резервируется, пользователь уведомлён
- Создал аккаунт после покупки — заказ привязывается автоматически
- Потерял письмо — может найти билет по номеру заказа + email (UC-V-02)

#### Путь B: С аккаунтом

**Основной поток:**
1. Нажимает «Купить билет»
2. Авторизуется (email+пароль или VK OAuth2) или регистрируется
3. Выбирает тип и дату
4. Вводит промокод при наличии
5. Переходит к оплате
6. Завершает оплату
7. Билет появляется в личном кабинете, QR — на email

**Альтернативные потоки:**
- VK OAuth2 — шаг регистрации пропускается
- Промокод недействителен — поле подсвечивается с пояснением
- Оплата не прошла — билет не резервируется

### UC-V-06 · Возврат билета 

| | |
|---|---|
| **Актор** | Покупатель (с аккаунтом или без) |
| **Триггер** | Не может посетить мероприятие |
| **Предусловие** | Билет куплен, срок возврата не истёк (настраивается в системе) |
| **Постусловие** | Билет аннулирован, деньги возвращены |

#### Путь A: Без аккаунта

**Основной поток:**
1. Пишет в поддержку, указывает email и номер заказа
2. Оператор находит заказ в CRM
3. Проверяет срок возврата
4. Подтверждает возврат вручную в модуле «Билеты»
5. Инициируется возврат через платёжный шлюз
6. Пользователь получает email-подтверждение

**Альтернативные потоки:**
- Срок возврата истёк — оператор уведомляет, возврат отклонён
- Ошибка шлюза — бухгалтер обрабатывает вручную

#### Путь B: С аккаунтом

**Основной поток:**
1. Заходит в личный кабинет → «Мои билеты»
2. Нажимает «Вернуть билет»
3. Система проверяет срок возврата из настроек
4. Пользователь подтверждает возврат
5. Инициируется возврат через платёжный шлюз
6. Получает email-подтверждение

**Альтернативные потоки:**
- Срок истёк — кнопка недоступна, отображается причина
- Ошибка шлюза — бухгалтер обрабатывает вручную

### UC-T-02 · Выбор и покупка торгового места 

| | |
|---|---|
| **Актор** | Верифицированный торговец |
| **Триггер** | Хочет забронировать место |
| **Предусловие** | Аккредитация одобрена, карта опубликована |
| **Постусловие** | Место куплено, QR-код аккредитации получен |

**Основной поток:**
1. Открывает раздел «Выбор места»
2. Изучает интерактивную карту (свободные / заблокированные / занятые)
3. Кликает на свободное место — система блокирует его на 10 минут
4. Видит детали: размер, цена, соседи
5. Нажимает «Забронировать»
6. Вводит промокод при наличии
7. Оплачивает
8. Получает подтверждение с номером места и QR-кодом

**Альтернативные потоки:**
- Место заблокировано другим — показан таймер снятия блокировки
- Оплата не прошла — блокировка снимается через 10 минут
- Промокод недействителен — поле подсвечивается

**Альтернативные потоки:**
- Автобэкап по расписанию — не требует ручного запуска

## Android-приложение — Охрана

### UC-G-01 · Первый запуск и авторизация

| | |
|---|---|
| **Актор** | Сотрудник охраны |
| **Триггер** | Начало работы |
| **Предусловие** | Аккаунт создан IT-администратором, токен выдан |
| **Постусловие** | Приложение авторизовано и синхронизировано |

**Основной поток:**
1. Скачивает приложение по ссылке от IT-администратора
2. Вводит токен доступа
3. Приложение скачивает локальную копию БД билетов
4. Готово к сканированию

**Альтернативные потоки:**
- Токен недействителен — обращается к IT-администратору

### UC-G-02 · Верификация посетителя (онлайн)

| | |
|---|---|
| **Актор** | Сотрудник охраны |
| **Триггер** | Посетитель предъявляет билет |
| **Предусловие** | Устройство в сети |
| **Постусловие** | Факт прохода залогирован |

**Основной поток:**
1. Наводит камеру на QR-код
2. Приложение сверяет с глобальной БД
3. Зелёный экран (действителен) или красный (ошибка)
4. Факт сканирования записан с меткой времени

**Альтернативные потоки:**
- QR не читается — ввод вручную по номеру билета
- Билет не найден / аннулирован — красный экран с причиной
- Нет сети — переключение в офлайн-режим (UC-G-03)

### UC-G-03 · Верификация в офлайн-режиме 

| | |
|---|---|
| **Актор** | Сотрудник охраны |
| **Триггер** | Потеря сети |
| **Предусловие** | Локальная БД синхронизирована при подключении |
| **Постусловие** | Проход залогирован локально; синхронизируется при восстановлении сети |

**Основной поток:**
1. Приложение переходит в офлайн — показывает предупреждение
2. Сканирует QR — проверка по локальной копии БД
3. Зелёный / красный экран, как в онлайне
4. Событие пишется в локальную очередь
5. При восстановлении сети — очередь синхронизируется автоматически

**Альтернативные потоки:**
- Повторное сканирование одного билета — допустимо, билет является пропуском входа/выхода, каждый проход пишется отдельной записью в лог

- Аккредитация отозвана — красный экран с причиной

### UC-G-05 · Синхронизация при восстановлении сети

| | |
|---|---|
| **Актор** | Сотрудник охраны |
| **Триггер** | Сеть восстановилась |
| **Предусловие** | В локальной очереди есть несинхронизированные события |
| **Постусловие** | Все события залиты в глобальную БД |

**Основной поток:**
1. Приложение обнаруживает сеть
2. Автоматически отправляет очередь
3. Показывает уведомление: «Синхронизировано N событий»

**Альтернативные потоки:**
- Синхронизация не удалась — очередь сохраняется, повтор при следующем подключении
