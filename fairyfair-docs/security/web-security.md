# Защита от веб-атак

## CSRF

Механизм: **HMAC-SHA256 подписанный nonce**.

Сравнение токенов выполняется через `hmac.compare_digest()` — **constant-time** функция, исключающая timing-атаки.

> Большинство реализаций используют обычное сравнение `==`, которое уязвимо к timing-атакам: время выполнения зависит от позиции первого несовпадающего байта, что позволяет атакующему угадывать токен побайтово.

---

## Rate Limiting

Реализован через **Redis**. Лимиты гранулярные — каждый эндпоинт имеет независимый порог:

| Эндпоинт | Комментарий |
|----------|------------|
| Авторизация | Низкий порог — защита от brute force |
| Регистрация | Средний порог |
| Покупка билета | Отдельный лимит — защита от автоматизированных покупок |
| Прочие API | Стандартный порог |

---

## SQL-инъекции

Исключены на уровне архитектуры:

- ORM с параметризованными запросами во всех обращениях к БД
- Отсутствие `raw SQL` в кодовой базе
- Входные данные никогда не интерполируются в SQL-строки напрямую

---

## Валидация входных данных

Все API-эндпоинты используют **Pydantic v2** со строгой типизацией.

- Некорректный тип данных → отказ на уровне валидации, до бизнес-логики
- Неизвестные поля в запросе → игнорируются или отклоняются (в зависимости от конфигурации модели)

---

## API-документация в production

Swagger UI и ReDoc **отключены** в production-режиме (`DEBUG=False`).

Это исключает автоматическую разведку структуры API злоумышленниками.
